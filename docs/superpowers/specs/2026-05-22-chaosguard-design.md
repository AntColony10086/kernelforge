## 2. Concept

**Chaosguard** is a small runtime + test harness that sits between an MCP agent and its dependencies (LLM providers + MCP tool servers) and makes the agent's behavior under infrastructure failure **observable, testable, and graceful**.

**Positioning:** *Not* a coding agent (would invite Cursor / Claude Code / Codex comparison). *Not* a framework demo. The product is the **resilience layer** + **chaos test suite** + **scorecard** that any MCP agent should adopt. A coding task is used purely as the demo workload because it is the most judge-legible domain.

**One-line pitch:** *Chaosguard is CI for agent reliability — drop it in front of any MCP agent and see how it behaves when the LLM browns out, an MCP tool flaps, or a downstream API silently fails.*

**In scope (MVP shipped before 2026-05-28):**
- Resilience runtime: LLM calls go through TrueFoundry AI Gateway with routing + fallback config; tool calls go through a wrapper that adds per-tool circuit breakers, result schema validation, and **side-effect verification for externally meaningful actions** via an `OutcomeLedger`; agent emits typed `DegradationEvent`s instead of hallucinating around dead tools; **final answers are rendered from the ledger, not freely generated by the LLM**.
- Chaos harness: scripted fault injectors that toggle LLM 5xx/timeout, MCP tool timeout, MCP tool error, and **silent-success-with-no-side-effect** (tool returns `{ok: true, pr_url: ...}` but the PR was never actually created).
- Naive baseline: the same agent code path run **without** Chaosguard (LLM calls go directly to raw OpenAI; tool calls go directly to MCP servers; no circuit breaker, no schema validation, no `DegradationEvent`, final answer is the planner's raw output with no truthfulness check), exposed to the same chaos scenario.
- Scorecard: per-run JSON + rendered table comparing naive vs resilient on task completion, honest reporting, retries, circuit-breaker activations, false-success rate, trace coverage.
- One demo workload (small Python repo + 1–2 hardcoded coding tasks) to drive the runtime end-to-end.
- Single-narrative demo video (~2 min 30 s) showing naive vs resilient side by side under the same scripted fault sequence.
- Public GitHub repo + Docker Compose for one-command repro.

**Explicitly out of scope:**
- LangGraph / agent framework — replaced by ~100-line Python state machine.
- Full OpenTelemetry stack + Jaeger UI — replaced by structured JSON traces + TrueFoundry's own observability.
- Pluggable policy DSL (mentioned in initial draft) — defer to "what's next".
- General-purpose coding agent — only the 1–2 hardcoded tasks needed for the demo.
- Multiple separate demo videos — one continuous narrative.
- Web dashboard — scorecard rendered as a static HTML/Markdown table.

**Component boundaries:**

| Component | Owns | Does not own |
| --- | --- | --- |
| TrueFoundry AI Gateway | LLM provider routing, retry, fallback, observability of LLM calls | Agent state, tool calls |
| TrueFoundry MCP Gateway | MCP server registry, auth, transport | Tool-level circuit breaking, degradation policy |
| Chaosguard resilience layer | Per-tool circuit breaker, schema validation, timeout enforcement, typed DegradationEvent emission, trace stitching | Provider routing (defers to AI Gateway) |
| Chaosguard agent runtime | Planner + executor state machine | Resilience logic (delegates to the layer) |
| Chaos harness | Fault injection middleware (LLM + tool side) + scenario scripts | Scoring logic |
| Scorecard generator | JSON-in → table-out comparison | Anything live; runs after a scenario completes |

**Critical division of labor** (per Codex round-1 review): **TrueFoundry owns LLM-level reliability primitives; Chaosguard owns agent-level behavior under tool failure.** The pitch deliberately does *not* compete with TrueFoundry; it composes with it.

## 4. Data flow

### 4.1 Successful step

`agent.plan(task)` → planner LLM call via `llm_client` → returns `next_action: tool_call(server, name, args)` → `tool_client.call(server, name, args)` → wrapper: check breaker state (closed) → forward to TrueFoundry MCP Gateway → tool returns JSON → validator passes → return result to executor → executor appends to scratchpad → loop until `next_action: finish`.

Every hop emits one trace event (timestamp, span_id, parent_id, kind, latency_ms, status, attrs). Traces flush to `traces/<run_id>.jsonl`.

### 4.2 LLM provider brownout

`llm_client.complete(...)` → POST to TrueFoundry AI Gateway → primary route (OpenAI) returns 503 → gateway's `retry_config` + `fallback_candidate` triggers backup route (Anthropic) → response returned to client with `x-tfy-routing` header noted in trace → executor continues. No code change inside Chaosguard. The win is configured in `routing_config.yaml`, not coded in agent logic.

### 4.3 Tool flap → circuit break + degradation

`tool_client.call("docs", "search", {q})` → wrapper checks breaker (closed) → forwards → 2× timeout (chaos middleware) → wrapper increments failure count, opens breaker after threshold (default 3) → on next call wrapper short-circuits with `ToolUnavailable("docs", "circuit_open", since_ts)` → executor catches → emits `DegradationEvent(step="search_docs", reason="circuit_open", partial_result=None, evidence_refs=[trace_id])` → planner gets degradation summary in next observation → planner adjusts plan ("skip docs lookup, use existing repo files for evidence") → task completes with caveats noted in final answer.

### 4.4 False-success prevention (the money shot)

`tool_client.call("github", "create_pr", {...})` → forwards → GitHub MCP returns `{"ok": true, "pr_url": "https://github.com/.../pull/42", "pr_id": 42}` — but the PR was never actually created (chaos middleware injects `silent_success_no_side_effect` — the MCP server's internal state shows no PR).

**Naive baseline** trusts the response, returns `"PR opened: https://github.com/.../pull/42 ✓"`. The judge can immediately verify the link is dead.

**Chaosguard** writes `open_pr: attempted → tool_returned` to the `OutcomeLedger` (not yet `verified_success`). The resilience layer then triggers a side-effect verification step: `tool_client.call("github", "get_pr", {"pr_id": 42})` → MCP returns 404 (or empty). Ledger transitions to `open_pr: unverified_success`. Planner receives a `DegradationEvent(step="open_pr", reason="side_effect_not_verified", partial_result={"claimed_pr_url": "...", "verification": "get_pr returned 404"}, evidence_refs=[trace_id_1, trace_id_2])`.

Final-answer renderer reads the ledger — it cannot say "PR opened" because `open_pr.state != verified_success`. It says: *"Patch ready (diff below). I attempted to open a PR; GitHub returned a URL but a verification call could not find the PR. The PR was likely NOT actually created. Inspect: https://github.com/.../pull/42 — verification 404."*

**This contrast is the demo's emotional peak.** Naive lies confidently; Chaosguard catches the silent failure by *verifying side effects*, not by checking error codes.

## 5. Component specifications

### 5.1 `llm_client`

- Single `complete(messages, **kwargs) -> Completion` method.
- Sends `X-TFY-METADATA: {"run_id": ..., "step": ..., "scenario": ...}` so TrueFoundry-side logs are joinable to our traces.
- No retry/fallback logic in this client; delegated to gateway. Surfaces gateway's `x-tfy-*` response headers into the trace.

### 5.2 `tool_client`

- `call(server, tool, args, timeout_ms) -> ToolResult | raises ToolError`.
- Maintains a `CircuitBreaker` per `(server, tool)` pair.
- States: `closed` (forward), `open` (short-circuit raise `ToolUnavailable` until cool-down), `half_open` (one trial call).
- Two named profiles. **Production default** (in code, the everyday config): 3 failures → open; 30 s cool-down; tool timeout 5 s. **Demo profile** (the one used in the recorded video, so the breaker opens visibly within the demo window): 2 failures → open; 8 s cool-down; tool timeout 1.2 s; half-open requires 1 successful trial to close. Both profiles live in `configs/breakers.toml`; demo recording sets `CHAOSGUARD_BREAKER_PROFILE=demo`.
- Validates result against a per-tool JSON schema (loaded from `schemas/<server>/<tool>.json`). Schema mismatch ≡ failure.
- Enforces a hard timeout via `asyncio.wait_for`.
- Every call emits one trace event with `breaker_state_before`, `breaker_state_after`, `validator_result`, `latency_ms`.

### 5.3 `DegradationEvent`

```python
@dataclass(frozen=True)
class DegradationEvent:
    step: str               # logical step name, e.g. "open_pr"
    reason: str             # "circuit_open" | "timeout" | "schema_invalid" | "tool_error" | "llm_unavailable"
    partial_result: Any | None
    evidence_refs: list[str]  # trace_ids supporting the claim
    timestamp_ms: int
```

Emitted by `tool_client` (and `llm_client` if the gateway itself fails — rare). Surfaced to the planner inside the next observation. Used by `final_answer` builder to forbid false claims about failed steps.

### 5.4 Agent state machine

~100 lines of Python. States: `PLANNING → EXECUTING → OBSERVING → (loop) | FINISHING`. No framework. Each transition writes one trace event.

### 5.5 Chaos middleware

Two pieces, both implemented as tiny FastAPI reverse proxies (not gateway-config injection — proxies are more deterministic and easier to film):
- `chaos_llm_proxy`: in front of TrueFoundry AI Gateway. Toggles 503 / 429 / timeout / brownout on the primary route per `chaos.toml`.
- `chaos_mcp_proxy`: in front of designated MCP servers. Toggles timeout / 500 / malformed-JSON-output / silent-success-with-fake-data per `chaos.toml`.

`chaos.toml` syntax (sketch):
```toml
[scenario.brownout_then_flap]
[[scenario.brownout_then_flap.faults]]
target = "llm:openai"
mode = "503"
duration_ms = 4000
start_at_step = 1
[[scenario.brownout_then_flap.faults]]
target = "mcp:docs"
mode = "timeout"
duration_ms = 8000
start_at_step = 3
```

### 5.6 Scorecard

Reads `traces/<run_id>.jsonl` + `outcome_ledger/<run_id>.jsonl` + ground truth ("did the PR actually exist on the MCP server's side?"). Emits a **4-row table** designed to persuade a sponsor judge in a 5-second on-screen flash:

| Metric | Naive | Chaosguard |
| --- | --- | --- |
| Useful result delivered | No / partial | Patch + tests |
| False success claims | 1 | 0 |
| LLM failover via TrueFoundry AI Gateway | No | OpenAI → Anthropic |
| MCP outage handling | Hammered / trusted bad result | Breaker + verified degradation |

Rendered as Markdown table + a tiny HTML page (`scorecard.html`). A fuller README-only scorecard keeps the detailed counters (retries, breaker opens, latency, etc.) for engineering credibility, but **only the 4-row version appears in the demo video**.

### 5.7 `OutcomeLedger`

A per-run append-only log of externally meaningful actions and their verified state. The final answer is rendered from this ledger; the LLM may *summarize* the ledger but cannot *invent* claims about completed actions.

States (per action):

```text
attempted -> tool_returned -> verified_success     # ledger trusts tool result only after side-effect check
attempted -> tool_returned -> unverified_success   # tool said success, side-effect check did not confirm
attempted -> failed                                # tool error / timeout / circuit-open
```

Schema:

```python
@dataclass(frozen=True)
class LedgerEntry:
    action: str               # "open_pr", "merge_pr", "run_tests", "write_file", ...
    state: Literal["attempted", "tool_returned", "verified_success", "unverified_success", "failed"]
    tool_response: dict | None
    verification: dict | None  # e.g., {"check": "get_pr", "result": 404}
    evidence_refs: list[str]   # trace_ids
    timestamp_ms: int
```

Verification policy is per-action and lives in `configs/verification.toml`:

```toml
[open_pr]
verify_with = "get_pr"
verify_args = { pr_id = "$tool_response.pr_id" }
verify_pass_condition = "response.status == 200 && response.body.state in ['open', 'closed']"

[run_tests]
verify_with = "read_file"
verify_args = { path = ".chaosguard/last_test_output.txt" }
verify_pass_condition = "exists && contains('passed')"
```

The final-answer renderer is a templated function. Pseudo-code:

```python
def render_final_answer(ledger: list[LedgerEntry]) -> str:
    parts = []
    for entry in ledger:
        if entry.state == "verified_success":
            parts.append(f"{entry.action}: confirmed ({entry.verification})")
        elif entry.state == "unverified_success":
            parts.append(f"{entry.action}: tool claimed success but verification failed ({entry.verification}). Treat as NOT done.")
        elif entry.state == "failed":
            parts.append(f"{entry.action}: failed ({entry.tool_response.get('error')}). Skipped.")
    return summarize(parts)  # LLM call constrained to a structured-output schema; cannot add facts not in `parts`
```

The LLM-side summarize call uses structured-output mode (JSON schema enforced) and a system prompt that explicitly forbids adding any completion claim not present in the input. This is the deepest defense against false success.

## 8. Tech stack

| Layer | Choice | Reason |
| --- | --- | --- |
| Language | Python 3.11 | fastest to ship, matches MCP SDK, matches TrueFoundry docs examples |
| LLM transport | TrueFoundry AI Gateway (OpenAI-compatible API) | sponsor scoring + does the routing/fallback for us |
| MCP transport | TrueFoundry MCP Gateway | sponsor scoring + central registry |
| Agent | hand-rolled ~100-line state machine | no framework learning curve; ships in 1 day |
| Resilience layer | hand-rolled | this IS the product, no shortcut |
| Chaos middleware | tiny FastAPI proxies + a toml config | deterministic faults, easy to film |
| Traces | structured JSON Lines + a tiny rendering script | no Jaeger/OTEL collector setup |
| Scorecard | Python script → Markdown + static HTML | no dashboard build |
| Packaging | Docker Compose (chaosguard + naive + TF gateway pointer + 3 MCP servers + chaos proxies) | one-command repro for judges |
| CI | one GitHub Actions workflow that runs the scenario and uploads the scorecard | proves the runtime, doubles as "Progress" evidence |

Things we are NOT using and explicitly rejected: LangGraph, Jaeger, React dashboard, Streamlit, Selectize tag UI, Postgres, vector DB, RAG, fine-tuning.
